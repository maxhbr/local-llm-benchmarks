Running **Aider benchmarks** locally against a locally deployed AI model is an excellent way to evaluate its agentic coding, reasoning, and instruction-following abilities.

Because Aider's benchmarking suite runs unreviewed code generated by the LLM (which could execute unintended or destructive commands like `rm -rf`), it is designed to run inside a isolated **Docker container** while communicating with your local model server.

Here is a comprehensive local benchmarking plan.

---

## 📋 Architectural Overview

To do this entirely on your machine, you will run a dual-layer setup:

1. **The LLM Host (Host Machine):** Your local inference engine (e.g., `llama.cpp`, Ollama, vLLM, or LM Studio) serving an OpenAI-compatible API.
2. **The Benchmark Runner (Docker Container):** Aider’s official benchmarking docker image that executes the tests safely and isolation-tests the code edits.

---

## 🛠️ Step 1: Set Up and Serve the Local Model

You must serve your local model using an API server that exposes an OpenAI-compliant endpoint (`/v1/chat/completions`). Ensure the server binds to `0.0.0.0` so the Docker container can communicate with it.

### Example: Serving via `llama.cpp` (Recommended for raw control)

```powershell
# Adjust your paths, context windows, and GPU offloading as needed
./llama-server --model ./models/Qwen3.5-27B-Coding-Q5_K_M.gguf \
               --ctx-size 32768 \
               --host 0.0.0.0 \
               --port 8080

```

> ⚠️ **Important:** Ensure your context window (`--ctx-size`) is set to at least **16k to 32k**. Benchmarking tasks send system prompts, coding instructions, and file historical data that will quickly overwhelm a default 2k or 4k context window.

---

## 📂 Step 2: Clone Aider & Benchmark Test Suites

On your host machine, pull down Aider’s core repository and its Polyglot coding benchmark framework (built on top of standard Exercism coding problems).

```bash
# 1. Clone Aider's primary repository
git clone https://github.com/Aider-AI/aider.git
cd aider

# 2. Create the benchmarking directory
mkdir tmp.benchmarks

# 3. Clone the polyglot dataset (includes exercises in Python, JS, Go, Rust, C++, etc.)
git clone https://github.com/Aider-AI/polyglot-benchmark tmp.benchmarks/polyglot-benchmark

```

---

## 🐳 Step 3: Launch the Sandboxed Container

Aider includes built-in scripts to handle Docker orchestration.

1. **Build the container** (Run once):
```bash
./benchmark/docker_build.sh

```


2. **Launch the container**:
```bash
./benchmark/docker.sh

```



Once inside the container's bash shell, install your cloned version of Aider in editable mode so it uses the environment natively:

```bash
# Inside the Docker container
pip install -e .[dev]

```

Verify that the container can successfully reach your host machine's LLM API:

```bash
curl http://host.docker.internal:8080/v1/models

```

*(If you are running Linux without Docker Desktop, replace `host.docker.internal` with your host's actual local bridge IP, e.g., `172.17.0.1`).*

---

## 🚀 Step 4: Run the Benchmark

Now, use the `benchmark.py` script. When testing local models, it is highly recommended to start with the `whole` edit format (where the model rewrites the entire file block) rather than `diff`, as weaker or smaller local models frequently fail the strict markdown search/replace syntax of diffs.

```bash
./benchmark/benchmark.py my-local-model-run \
  --model openai/your-model-name \
  --openai-api-base http://host.docker.internal:8080/v1 \
  --openai-api-key "dummy-key" \
  --edit-format whole \
  --exercises-dir polyglot-benchmark \
  --threads 1 \
  --num-tests 5

```

### 🔍 Key Flag Explanations:

* `--model`: Prefixed with `openai/` so Aider routes the schema appropriately to your local API.
* `--threads 1`: Keep threads at **1** initially for local models. If your GPU and inference framework support parallel batch processing (like vLLM), you can increase this to speed things up.
* `--num-tests 5`: **Crucial for testing.** Run a micro-batch of 5 exercises first to ensure the API responses are mapping correctly before committing to all 200+ tests.

---

## 📊 Step 5: Analyze the Results

Aider will output a new timestamped results directory inside `tmp.benchmarks/`.

To compile the results and see how your model scored, run Aider's evaluation summary utility inside the container:

```bash
./benchmark/report.py tmp.benchmarks/YYYY-MM-DD-HH-MM-SS--my-local-model-run

```

This will output a clean breakdown showing:

* **Pass Rate (Try 1):** Percentage of assignments the model completed perfectly on its first attempt.
* **Pass Rate (Try 2):** Percentage of tasks solved after Aider automatically fed back compilation/unit-test errors to the model (a major indicator of *agentic correction*).

---

## 💡 Troubleshooting & Local Optimization Tips

* **Temperature control:** Aider forces its own temperature settings on benchmarks by default. If your model requires highly specific sampling parameters (like a strict `0.0` or custom Top-P for reasoning models), create a `.custom-model-settings.yml` file in your runtime directory to lock those in.
* **The Model "Mangles" Code Blocks:** If your report shows 0% because the model cannot format its edits correctly, try changing `--edit-format whole` to `--edit-format architecture-dependent` options, or try a model specifically fine-tuned for tool use.
* **System Crashes / Out of Memory (OOM):** Running the Python tests alongside local heavy inference on one consumer GPU can trigger CUDA OOM. If this happens, offload more layers to your CPU or use a lower quantization variant (e.g., `Q4_K_M` or `Q5_K_M`).
